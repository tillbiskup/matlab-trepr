Why writing a GUI?

A rather lengthly answer to a simple question

A bit of history

Originally, the trEPR toolbox started as a collection of scripts, soon functions, that did their job for the first couple of years for the author's purposes of standard data processing and data analysis of trEPR data.

But with the years there was some demand in having a nice, relatively easy-to-use graphical users interface (GUI) for all those functions, as it was no longer easy for others than its author to get an overview of all the functions that were provided by the toolbox, given that there was never a really well-written documentation.

Due to constant encouragement and support from a good friend and colleague (for details open the About window and wait a bit) the development of the GUI continued beyond initial stages that contained, while usable with some data, multitudinous bugs.

The current GUI: a complete redesign of GUI and toolbox

The current version of the GUI is a complete rewrite of the original one, having both modularity and stability in mind from the very beginning.

The main goals of this redesign of the GUI were, besides improvements on the coding, a reduction in size so that it fits even to such small monitors as that of the smallest MacBook Pro with a resolution of only 1280x800 px - the machine currently in use by its main developer.

The final aim of the GUI is to provide the user with an easy-to-use and robust interface for the daily data processing and data analysis of trEPR data. Key concepts are the import of a variety of data formats and export capabilities both of figures and data for further processing with other, more specialised, types of software (including, but not restricted to, MATLAB™).

Will there still be a command line version?

The short answer: Yes, every real functionality of the toolbox is anyway completely separated from the GUI. That means that you still can and will always be able to perform all the tasks you wish on the Matlab™ command line.

Furthermore, and what might be even better: There is now a new command line (CMD) within the GUI that is still pretty much under constant development and will eventually make it possible to completely "script" the GUI.

Besides that, many parts of the GUI can be controlled in part or entirely via keyboard, therefore minimising the necessary use of the mouse (pointer device). It still holds true: If you know the keyboard shortcuts, you're much faster.
