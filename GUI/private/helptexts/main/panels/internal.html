GUI Internals

This panel provides access to different GUI interna and their settings.

The controls

A short description of all the controls of the panel.

GUI directories

Show/set directories used by the GUI to load/save data, figures, etc.

The Show button opens a window that lets you inspect and change the directories the GUI stores internally for loading/saving data, figures, and other things.

Messages/Log levels

Set message and log levels.

The debug level setting affects which messages are displayed in the status window.

The messages setting will have some effect in the future.

Snapshots

Save/load snapshots. Snapshots are saved states of the trEPR GUI. They are saved as MATLAB™ MAT-files in a directory that can be configured using the "GUI directories" settings described above.

Warning: Loading snapshots will overwrite the current state of the GUI. Currently, there is no way to undo this. Therefore, it is save to first save the current state and then load another state.

The button Show opens an additional window letting you change the directory for snapshots, display information about saved snapshots, and load a selected snapshot.

Command line (CMD)

The command line (CMD) is a very powerful feature of the trEPR GUI. It lets you control (eventually) all features of the GUI via the builtin command line and as a consequence via scripts.

For an introduction to the command line (CMD), see the builtin help for the CMD.

With the Save history switch you can control whether the cmd history gets saved on the file system (and is therefore persistent between trEPR GUI sessions). The default location for the history file is "~/.trepr/history". This can be changed via the configuration.

The Show history button opens a separate window displaying the history and letting you evaluate selected commands from the history.

And the Run script button opens a file selection dialogue box where you can select a script file to be executed within the trEPR GUI. Please note that until now there is no automatic code checking for script files. Therefore, the parser will try to execute the script line by line.
